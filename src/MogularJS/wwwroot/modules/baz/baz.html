Baz => Application Modularity

Create a web application having multiple SPAs in order to manage complexity by having smaller manageable SPAs, allow technology variance based on features in a SPA, enable multiple development teams to build the application, etc. (an ecommerce application can be compose of seperate SPAs for product browsing, account management and checkout, developed by 3 teams, having specialized application architectures and technology stacks) .

MogularJS composes Foo and Bar SPAs which are mutually exclusive (only one of them is accessed at a time) and User SPA which is accessible along with the other SPAs.

Angular's ng-app directive allows bootstrapping of only first application found on DOM, however angular.bootstrap() can be used to bootstrap multiple applications. When SPAs work in conjunction, they can communicate with each other thorough shared services by maintaining service instance globally.

Foo and User SPAs consume MessagingService to update/display User Name. The service instance in managed by MessagingServiceFactory as a window global variable.

(updated user name displays on the right in menu)

When creating multiple SPA, set base header tag properly in the template so that links within each SPA resolves correctly. Menu links on the left do not have SPA name in their href but the links resolve to contain SPA name as it is set in the base header tag.

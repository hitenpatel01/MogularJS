Quux => Module-level Modularity

Every application feature in MogularJS is a individual module, created as a folder and comprising of everything required for the feature like controllers, services, directives, angular components, models, templates, configuration and even unit tests. 

\baz                            \messaging                          \corge
    \baz.config.json                \messaging.config.json              \corge.config.json
    \baz.config.ts                  \messaging.config.ts                \corge.config.ts
    \baz.controller.ts              \messaging.model.ts                 \corge.controller.ts
    \baz.html                       \messaging.module.ts                \corge.html
    \baz.module.ts                  \messaging.service.ts               \corge.module.ts
    \baz.service.ts                                                     \corge.service.ts
                                                                        \corge-detail.controller.ts
                                                                        \corge-master.controller.ts

Modules contain only the necessary file types required to build the feature and follow naming convention <name>.<type>.<extension> instead of sub folders to avoid unnecessary folder traversal, however they could be used if code is better organized for a particular module.

Modules can be categorized into following types:

  • Root Modules: entry point of a Single Page Application where service behaviors (ex: routing) are configured.
    ex: Foo, Bar and User
  • Feature Modules: unique application feature that could be dependent on shared modules.
    ex: Baz, Quux, Corge, Grault, Garply, Waldo and Fred
  • Shared Modules: reusable components and services (ex: menu, messaging, notification, etc).
    ex: Menu, Messaging and Notification
  • Core Module: infrastructural components/services, base classes and services implementing cross cutting concerns.
    ex: Core
